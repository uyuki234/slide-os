# 自分の回答
1.
make runを打つと、Makefileに記述されている内容が実行される
まず、
run: kernel.elf
	qemu-system-i386 -kernel kernel.elf
で、kernel.elfを作成しようとする

kernel.elf: boot.o kernel.o linker.ld
	$(CC) -T linker.ld -ffreestanding -nostdlib -o $@ boot.o kernel.o -lgcc
kernel.elfの依存関係はこの通りで、boot.o kernel.o linker.ldが必要。
ファイルが揃ったら、$(CC) -T linker.ld -ffreestanding -nostdlib -o $@ boot.o kernel.o -lgccが実行される（←のコマンドの意味はあんまりわかってない。）

boot.o: boot.s
	nasm -f elf32 $< -o $@
boot.oはboot.sから作る

kernel.o: kernel.c
	$(CC) $(CFLAGS) -c $< -o $@
kernel.oはkernel.cをもとに作成

# Fableによる採点、講評
採点しました。結論: 1・3・5はほぼ合格、2・4・6が要修正です。LT本番で突っ込まれると危ないのは2と6なので、そこを重点的に直します。

Q1: ◎(合格) Makefileの依存関係の連鎖を正確に追えています。これはビルドの説明として十分です。残った「意味があんまりわかってない」リンクコマンドだけ埋めましょう:

-T linker.ld — 「配置はこの指示書に従え」
-nostdlib — 「libcやcrt0(普段main()の前に走るアレ)をリンクするな」。普段は黙って混ぜられているものを、明示的に断っているのがポイント
-o $@ — $@ はMakefileの自動変数で「ターゲット名」(=kernel.elf)。$< は「最初の依存ファイル」
-lgcc — コンパイラ用の補助ルーチン集(保険)。ここは「今回は理解しきらなくてよい」枠のままでOK

つまりこの1行は「boot.oとkernel.oを、linker.ldの地図に従って、OSの部品を一切混ぜずに結合しろ」です。
Q2: ✗(ここが一番危ない) 「qemuの上にprintfが存在しない」は2つの点でずれています。
まず、QEMUは無関係です。QEMUはただの仮想PCで、その上でLinuxを起動すればprintfは普通に動きます。「QEMU上だから使えない」と言うと、メモ16節の誤解2(QEMU上なのでOSではない)と混線した理解に聞こえます。
正しい構図はこうです: printfはlibc(標準ライブラリ)の関数で、しかもその中身は最終的にOSのシステムコール(write)を呼んで画面出力を頼んでいます。今回の環境には (a) libcがリンクされていない(-nostdlib)、(b) そもそも頼む先のOSが存在しない、の二重の不在があります。実験1で見た「stdio.hが存在しない」エラーが(a)の証拠です。
LT用の模範解答: 「printfは標準ライブラリの関数で、裏でOSに画面出力を頼んでいます。今回はそのライブラリもOSも存在しない場所でコードが動いているので、printfは『禁止』ではなく『存在しない』んです」
Q3: ○(方向は合格、一言足す) 「Cを動かす前に必要な動作」は正しい。ただLTで「必要な動作って何?」と聞かれたら答えられますか? 中身は実質1つで、espにスタックの上端を設定することです。「Cの関数はローカル変数も戻り番地も全部スタックに置くので、スタックがないとCは1行も動けない。それを用意してからkmainを呼ぶのがboot.s」まで言えれば完璧です。あとMultibootヘッダ(QEMUへの合言葉)を置く仕事もありましたね。
Q4: △(半分) アドレスであることは正解。「VGAテキストバッファが何かわかってない」と自己申告してくれたのが良いです。埋めましょう:
VGAは昔のPCの画面表示規格で、そのテキストモードでは画面装置(ビデオカード)が0xB8000〜のメモリ領域を常に監視していて、そこに置かれた値をそのまま80×25の文字として描画し続けている、という約束になっています。つまりあの領域は「普通のメモリ」ではなく、画面という装置に直結した窓です。だから代入=表示になる。これをメモリマップドI/Oと呼ぶ、まで言えればLTで一番おいしい部分が完成します。
「なぜ代入で文字が出るの?」→「その番地は画面装置が見張っている特別な領域だから」が一言版です。
Q5: ◎(合格) 区別も「変換はOSの仕事」も正確です。LTでは実物を添えると強い: 「Nキーを離すとB1が届く、押した/離したの区別まで生で見える世界」のスクショがそのまま証拠になります。
Q6: △(前半○、後半が混乱) 「ブートローダーの役割」は正解。ただし「本物のPCなら誰がやるか」の答えは「GRUBなどのブートローダ」です(BIOSがGRUBを読み、GRUBがカーネルを1MBに配置して_startへ飛ぶ——QEMUの-kernelはこのGRUBの部分を代行)。
「USBにboot.asmを保存して刺す」は2点修正が必要です: (a) 刺すのはソースコード(.asm)ではなくビルド済みバイナリ(boot.bin)、(b) それはStep 1のブートセクタ実験の話であって、slide-os本体(Multiboot方式)はその方法では起動しません——先頭512バイトに収まっていないし、誰かがELFを解釈して1MBに置く必要があるからです。本体を実機で動かすならUSBにGRUBを入れてkernel.elfを読ませる構成になります。2つの実験が頭の中で混ざっているので、LTでは「ブートセクタ実験(BIOS直起動)」と「本体(Multiboot、ローダに任せた)」を明示的に分けて話すと事故りません。
Q7: ○(態度は正解、言い切りの型を持つ) 迷いを見せるより、準備した答えを言い切る方が誠実に聞こえます。メモ16節ベースの模範解答: 「一般的なOSが持つプロセス管理やファイルシステムはないので、OSの入口に触れる最小デモと呼ぶのが正確です。ただ、Linux上のプログラムではなく、起動時に直接実行される自作カーネルではあります」。「できないことを先に自分から言う」のが最強の防御です。

総評
自力で説明できた範囲(ビルドの流れ、boot.sの位置づけ、スキャンコード)と、埋めるべき穴(printfの二重の不在、0xB8000の正体、GRUBの位置)がきれいに分離できました。この3つの穴は今日の説明で埋まったはずなので、Q2・Q4・Q6だけ、明日の本番前にもう一度自分の言葉で言い直してみてください(書かなくても、口頭で一人リハで十分です)。
特にQ2の模範解答の一文は、LTのSlide 2(printfはOSに乗っている)の台本そのものとして使えます。
