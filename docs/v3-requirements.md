## 1. ゴール

**「GUI操作ができるOS」にする。**
v2のテキストセル・ウィンドウシステムをピクセルグラフィックスに移植し、
マウスで直接操作できるデスクトップとして成立させる。

### 成功条件(これが全部できたらv3完了)

1. GRUBのISO起動で 1024x768x32bpp のフレームバッファを取得し、描画できる
2. ウィンドウがピクセル描画(タイトルバー・枠・背景)され、**マウスドラッグで移動できる**
3. **クリックでフォーカスが切り替わる**(v2のTab/矢印操作も併存させ、マウス撤退時の保険にする)
4. v2のアプリ(home / slides / terminal)がビットマップフォント上で動作する
   (スライドが動き続けること。slide-osの名を維持する)
5. シリアル(COM1)にカーネルログが出る(グラフィックス時代のデバッグ手段)

### 非ゴール(やらないこと)

- ネットワーク、ブラウザ、動画再生(供養済み。スコープ外を維持)
- マルチプロセス/スケジューラ(v4以降の候補として温存)
- x86_64 / UEFI 移行(下記「決定の記録」参照。これもv4以降の独立テーマとして温存)
- アンチエイリアス、アルファブレンド等のリッチ描画

## 2. 決定の記録: ブート方式は案a(GRUB + Multiboot、i386維持)

3案の比較結果と決定理由。後から「なぜ案bにしなかったのか」を思い出すための記録。

| 観点 | 案a: GRUB+Multiboot | 案b: UEFI+GOP(x86_64) | 案c: Bochs VBE直叩き |
|---|---|---|---|
| 環境構築 | 軽い(brew数個) | FWはQEMU同梱だがPE/COFF用ツールチェーンが別途必要 | ゼロ |
| v2資産 | ほぼ100%再利用 | 割り込み基盤を64bitで全面書き直し+ページング必須 | 100%再利用 |
| 新規学習 | 本物のブートローダ、Multiboot info回収 | ロングモード、ページング、UEFI(最大) | PCI列挙(QEMU固有) |
| みかん本接続 | 弱い | 直結 | なし |

**決定理由:**

- v3のゴールはピクセルGUIであり、ブート方式の刷新ではない。案bは最初のピクセル
  を打つ前にロングモード移行という大工事(実質1バージョン分)が挟まる。
- v3本体(描画・ウィンドウ・マウス・malloc)はアーキテクチャ非依存の知識なので、
  後でUEFI/x86_64に移行しても持っていける。案b相当は「v4以降の移行編」として温存。
- 事実確認済み(2026-07-06): `i686-elf-grub` はHomebrew公式formulaに存在し、
  ソースビルド不要。ドラフト時点の「grub-mkrescue環境構築が要検証」の懸念は解消見込み。
- 案aの隠れた教材価値: v1/v2はGRUBが `ebx` に渡すMultiboot info構造体を無視してきた。
  v3ではフレームバッファ情報(M1)とメモリマップ(M5)で2回これを回収する。
  「ブートローダ=カーネルへの情報の引き継ぎ役」という欠けていたピースが埋まる。

## 3. 技術的前提・既知の罠(着手前に共有しておくもの)

- **グラフィックスモードに入ると 0xB8000 のVGAテキストは消える。**
  v2のkprintfは盲目になる。→ シリアル出力(M0)を何よりも先に作る。
- **QEMUの `-kernel` 直接ロードはMultibootヘッダのvideo要求を無視する。**
  フレームバッファが欲しければGRUB経由(ISO起動)が必須。これが `make run-iso` を
  作る理由そのもの。移行後のカーネルは「mbinfoにフレームバッファ情報がない」場合を
  検出してシリアルにエラーを吐いて止まる(防御的な起動チェックの練習台)。
- Multiboot1でのフレームバッファ要求: ヘッダflagsのbit 2を立て、
  mode_type=0(リニアグラフィックス)、width/height/depth を指定する。
  GRUBは応答としてinfo構造体のflags bit 12を立て、framebuffer_addr / pitch /
  width / height / bpp を返す。**pitch(1行のバイト数)は width×4 と一致すると
  限らない**。putpixelは必ずpitchで計算する。
- バックバッファは 1024×768×4B ≈ 3MiB。静的配列(BSS)で持つ。QEMUのデフォルト
  RAM(128MiB)では問題にならない。ウィンドウ個別のピクセルバッファも当面は
  静的プール(v2方式)で持ち、動的化はM5まで遅延する。
- 8x16ビットマップフォントは自作せず、パブリックドメイン相当の既存フォントを
  Cの配列に変換して使う(出典とライセンスは組み込み時に必ず確認して記録する)。

## 4. モジュール分割

実装順。各モジュールで「実装 → 動作確認 → 確認実験 → git commit」のサイクルを回す。

| # | モジュール | 内容 | 主要概念 |
|---|---|---|---|
| 0 | シリアル出力 | COM1(0x3F8)へkprintfを複製、`-serial stdio` | UART、グラフィックス時代のprintfデバッグ |
| 1 | ブート移行 | grub.cfg、ISOターゲット、Multibootヘッダvideo要求、ebx回収 | ブートローダの実像、info構造体、ISO(El Torito) |
| 2 | 描画プリミティブ | putpixel、fill_rect、8x16フォント描画、gputs | フレームバッファ=pitch付き2次元配列、ビットマップフォント |
| 3 | ウィンドウ移植 | バックバッファ合成・zorder・フォーカスをピクセル座標へ | v2設計の再利用性の答え合わせ |
| 4 | PS/2マウス | IRQ12、8042のAUXポート有効化、3バイトパケット、カーソル、ドラッグ | パケットのステートマシン、割り込みは受付係(v2の復習) |
| 5 | メモリ管理 | Multibootメモリマップ→物理ページアロケータ→簡易malloc | 静的プール制約の解消、フリーリスト |
| 6 | ストレッチ | ダメージ矩形、ダブルクリック抽象化、日本語フォント | 最適化は計測とセット |

### 各モジュールの完了条件と確認実験(候補)

**M0 シリアル出力**
- 完了条件: `make run` でターミナルに起動ログが流れる。kprintfと同じ書式が使える。
- 確認実験: LSR(ラインステータス)の送信可能ビット待ちを消すとどうなるか。
  ボーレート除数を壊すと文字化けするか(QEMUはどこまで再現するか)。

**M1 ブート移行**
- 完了条件: `make run-iso` でGRUBメニュー経由で起動し、mbinfoのフレームバッファ
  情報(addr/pitch/解像度/bpp)がシリアルに出力される。画面全体を単色で塗れる。
- 確認実験: video要求フラグを消すとどうなるか。`-kernel` 起動で防御チェックが
  働くか。チェックサムを壊すとGRUBは何と言うか(v1のMultiboot実験の再現)。
- 撤退ライン: **環境構築(grub-mkrescue/xorriso)で半日以上溶けたら案cに切り替え、
  描画(M2)から着手する。** 案cならM2以降は全て共通。

**M2 描画プリミティブ**
- 完了条件: 矩形と文字列が任意座標に描ける。画面クリアがある。
- 確認実験: pitchをwidth×4の決め打ちに変えて表示が壊れるか(壊れなくても
  「たまたま一致しているだけ」を確認する)。フォント配列を1ビットずらす。

**M3 ウィンドウ移植**
- 完了条件: v2の6枚プール・zorder・Tab切替がピクセル描画で動き、アプリ3本が
  ビットマップフォントで表示される。タイトルバーと枠がある。
- 確認実験: クリッピングを消して画面外に窓を動かす(v2と同じ壊し方が通用するか)。

**M4 PS/2マウス**
- 完了条件: カーソルが動く。左クリックでフォーカス切替、タイトルバードラッグで
  ウィンドウ移動。
- 確認実験: パケット同期を1バイトずらすとカーソルが暴れることを確認(ステート
  マシンの存在意義)。EOIを消す実験の再現(IRQ12はスレーブPIC側=EOIが2回必要)。
- 撤退ライン: **難航したらキーボード操作のみのGUIで成立させる**(成功条件3の
  Tab/矢印併存はこのための保険)。

**M5 メモリ管理**
- 完了条件: mbinfoのメモリマップをシリアルにダンプできる。ページアロケータと
  kmalloc/kfreeが動き、ウィンドウを動的に生成・破棄できる。
- 確認実験: 二重freeや解放後使用を意図的にやって何が起きるか観察する
  (「なぜ実世界のmallocは壊れ方が分かりにくいか」の入口)。
- 撤退ライン: **難航したら静的プール継続でv3は成立とする**(M3/M4はM5に依存しない)。

**M6 ストレッチ(任意)**
- ダメージ矩形は「全画面再描画のtick消費を計測してから」着手する。計測なしの
  最適化はやらない。

## 5. 正直ポイントの予告(v2から引き継ぐ手抜き+v3で増える手抜き)

- リングバッファの競合対策なし(シングルコア前提)は継続。マウスでも同じ手を使う。
- 毎フレーム全画面再描画は当面継続(M6まで)。
- マウスカーソルは合成の最後に直接描く(ハードウェアカーソルなし)。
- スキャンコードセット1・USレイアウト・Ctrl/Alt未対応は継続。
- CPU例外は一律パニックのまま(回復なし)。
